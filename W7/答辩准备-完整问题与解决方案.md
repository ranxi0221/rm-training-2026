# 第一部分：基础概念（必须先读）

## 1.1 什么是 BGR？什么是 HSV？

### BGR —— 电脑怎么看颜色？

电脑里的彩色图像，每个像素点由三个数字组成：

```
B（Blue 蓝色）= 200
G（Green 绿色）= 150  
R（Red 红色）= 140
```

三个数字合在一起，就描述了这个像素是什么颜色。B=200 说明"蓝色成分很多"，R=140 说明"红色成分中等"，G=150 说明"绿色成分也中等"。所以这个像素看起来是**偏蓝的灰白色**。

就像调颜料：蓝色颜料放两勺，红色放一勺半，绿色放一勺半，搅在一起得到的就是蓝灰色。

> 为什么叫 BGR 不是 RGB？因为 OpenCV（我们用的图像处理库）习惯把蓝色放前面。只是名字不同，道理一样。

### HSV —— 换一种方式描述颜色

BGR 的问题是：你想描述"偏蓝的灰白色"，数字不好直观判断。HSV 换了个思路：

```
H（Hue 色相）     = 110°   → 这是"什么颜色"（0°红→120°绿→240°蓝）
S（Saturation 饱和）= 15     → 这是"颜色有多纯"（0=灰白，255=最鲜艳）
V（Value 亮度）    = 200    → 这是"有多亮"（0=纯黑，255=最亮）
```

- H=110 说明它在大致蓝色区域
- S=15 说明它颜色很淡、接近灰白（纯蓝色 S 会到 200+）
- V=200 说明它比较亮

**为什么搞两套？** 你用人话说"帮我找蓝色灯条"→ 用 HSV（H 在蓝色范围）。但过滤"白墙"→ 用 BGR（看蓝色是不是真的远超红绿色）。

## 1.2 什么是灯条？什么是装甲板？

RoboMaster 的装甲板长这样：

```
┌─────────────────────────────┐
│   [蓝LED灯条]    [蓝LED灯条]  │  ← 两根蓝色的发光灯条
│      ████          ████      │
│      ████    1     ████      │  ← 中间印着数字"1"
│      ████          ████      │
│   [蓝LED灯条]    [蓝LED灯条]  │
└─────────────────────────────┘
```

- **灯条**：两根竖着的 LED 发光管，我们第一要找的东西
- **装甲板**：整块板子，灯条配对之后框出来的区域
- **数字**：印在装甲板正中间，表示这是几号装甲板

### 检测流程（先大致感受一下）

```
相机拍到的画面 → 找出所有蓝色/红色的像素 → 
滤掉噪声 → 把像素团成"灯条" → 
找两根长得像的灯条配成一对 → 框出来就是"装甲板" →
从装甲板中间裁出数字区域 → AI模型判断数字是几
```

## 1.3 什么是形态学操作？

简单说：**把图像中的白色区域"修理"一下**。

### 开运算 — 去掉小白点

想象一张纸上有几个小黑点。你用橡皮擦一擦——小黑点没了，但大字还在。

```
原始：  ████ · ████        （· 是一个噪点小白点）
开运算： ████   ████        （噪点消失，大的白色块还在）
```

原理：先用橡皮轻轻擦（腐蚀），再描一遍边（膨胀）。
小点：擦没了 → 描不回来了 ❌
大块：擦边缘 → 描回来恢复 ✅

### 闭运算 — 填补小黑洞

```
原始：  ███  ███             （灯条中间断裂了）
闭运算： ████████             （断裂被填上）
```

原理：先描粗（膨胀），再擦回（腐蚀）。
小缝：描粗时被填上 → 擦回时还在 ✅

---

## 1.4 什么是 EMA 平滑？

EMA = 指数移动平均。用它来解决"黄框一直抖动"的问题。

```
没有 EMA：每一帧独立检测 → 框的位置每帧可能差几个像素 → 画面里框在抖
有 EMA：新位置 = 旧位置×0.4 + 当前检测位置×0.6
```

0.4 叫 `smooth_alpha`。数字越大 → 越平滑但越"跟不上"。我们最后定了 0.4，既不太抖也不太慢。

# 第二部分：我们的项目是怎么做的

## 2.1 整体流程（一张图看懂）

```
相机 → 图像翻正 → 降低亮度 → 转HSV →
找出蓝色像素 + 找出红色像素 →
用BGR验证这些像素真的是蓝/红（不是白墙）→
形态学去噪+补洞 → 把像素聚成"轮廓" →
筛选：面积够大/形状够瘦长 → 这就是"灯条" →
LED真伪验证（中心比边缘亮很多）→
配对：找两个尺寸相近、距离合适的同色灯条 →
每色只保留最大的一个装甲板 →
从装甲板中间裁出数字区域 → AI模型识别数字 →
黄框标出装甲板 + 文字标签（如 blue_1）
```

---

## 2.2 第一步：找到蓝色和红色的像素（HSV 分离）

**目的：** 从画面中挑出所有"看起来是蓝色"或"看起来是红色"的像素。

**方法：** 把 BGR 图像转成 HSV，然后设门槛：

```yaml
蓝色灯条：H在 100~125 之间（蓝紫色范围），S > 40（不能太灰），V > 150（要够亮）
红色灯条：H在 0~10 或 170~179，S > 30，V > 100
```

**为什么红色有两段（0~10 + 170~179）？** 因为在 HSV 的色环里，红色正好跨过 0° 这条线：红色在色环的"头"和"尾"都有。0~10 是深红，170~179 是浅红/粉红。

经过这一步，我们得到一个**黑白图（mask）**：白色 = 符合条件的像素，黑色 = 不符合的。

---

## 2.3 第二步：BGR 颜色纯度过滤（W7 最重要的改进！）

**问题：** 经过 HSV 分离后，白墙、灰色桌面也会被标记为"蓝色"。因为它们的 H 也刚好在 100~125 范围内，S 和 V 也碰巧满足条件。

**根因：** 白墙的 BGR 是 (B=180, G=150, R=140)。蓝色三个通道差不多一样强。而蓝灯条的 BGR 是 (B=240, G=80, R=50)。蓝色远远强于红色和绿色。

**解决：** 在 HSV 之后再加一道检查——

```
对于标记为"蓝色"的像素：
  ① B（蓝色值）必须 > 120                （够亮，不是暗噪点）
  ② B 必须 > R × 1.7                     （蓝色是红色的 1.7 倍以上）
  ③ B 必须 > G × 1.7                     （蓝色是绿色的 1.7 倍以上）
  三个条件全满足 → 是真蓝LED ✅
  任何一个不满足 → 是白墙/灰面 ❌ 扔掉！
```

**用表格直观感受：**

| 是什么 | B | R | B/R | 是否 > 1.7？ |
|------|:--:|:--:|:--:|:--:|
| 蓝LED中心 | 240 | 50 | 4.8 | ✅ 通过 |
| 蓝LED边缘 | 200 | 80 | 2.5 | ✅ 通过 |
| 白墙 | 180 | 140 | 1.28 | ❌ 拒绝 |
| 灰桌面 | 130 | 125 | 1.04 | ❌ 拒绝 |

**这就是为什么 BGR 过滤能根治误检！** 白墙的蓝色只是"稍微比红色多一点"，而真蓝LED的蓝色是"碾压级别"的超过红色和绿色。

---

## 2.4 第三步：形态学 + 找轮廓

HSV+BGR 过滤后，我们有一个黑白 mask。但上面可能还有一些小白点（噪点），灯条中间可能有些像素没被标记（空洞）。

- **闭运算（先膨胀再腐蚀）**：填补灯条内部的微小空洞
- **开运算（先腐蚀再膨胀）**：去除小白点噪点（暗环境关掉，会伤到细灯条）

然后 `findContours` —— 把白色的连续区域找出来，每个区域变成一个"轮廓"。

---

## 2.5 第四步：筛选灯条

不是每个轮廓都是灯条。我们要筛选：

```
① 面积在 20 ~ 50000 像素之间   → 太小=噪点，太大=不是灯条
② 宽高比 < 0.6                  → 灯条是又高又瘦的（宽/高 < 0.6）
                                 反光和噪点是比较"胖"的（宽/高接近1）
```

**为什么用宽高比？** 灯条是竖着的长条：宽 15px，高 80px → 宽高比 = 0.19。反光可能是 40×40 的方块 → 宽高比 = 1.0。0.6 这个门槛把方块全过滤掉了。

---

## 2.6 第五步：LED 真伪验证

**问题：** BGR 过滤 + 形状筛选后，还有极少数反光能蒙混过关。

**最后的防线：** 真 LED 和反光在亮度分布上根本不同。

```
真 LED：     反光：
  ░░░░         ░░░░
 ██████       ██████      
 ██████       ██████      真 LED 中心极亮，边缘暗
 ██████       ██████      反光整块差不多亮
  ░░░░         ░░░░
```

**验证方法（代码做的事）：**
```
取灯条中心 8×8 区域 → 找最亮像素的灰度值
取灯条边缘 24×24 区域 → 算平均灰度

必须同时满足：
  ① 中心最亮像素 > 180                         （确实很亮）
  ② 中心最亮像素 > 边缘平均灰度的 2.5 倍        （中心远亮于边缘）
  ③ 中心最亮像素 > 中心平均灰度的 1.5 倍        （中心是个尖锐亮点）
```

只有真正的 LED 发光体才有这种"中间极度亮、周围迅速暗"的特征。反光整块亮度均匀，第三条就挂了。

---

## 2.7 第六步：配对

现在我们有了一堆灯条。接下来把它们两两配对。

**怎么配对？** 两个灯条必须满足：

```
① 颜色相同（都是蓝或都是红）
② 水平距离在 5 ~ 2500 像素之间（不能太近也不能太远）
③ 垂直偏差 < 400 像素（几乎在一条水平线上）
④ 两个灯条的高度必须相近（小高度/大高度 > 0.4，不能一高一低）
⑤ 两个灯条的宽度必须相近（小宽度/大宽度 > 0.3，不能一胖一瘦）
```

⑤ 很关键：真正的两个灯条来自同一块装甲板，尺寸几乎一样。假配对往往是一个大真灯条 + 一个小噪点，尺寸一高一胖，直接被过滤。

**每色只保留一个装甲板：** 如果有多个蓝灯条配对出了好几个蓝色装甲板，只保留**面积最大**的那个（最大的说明灯条最大、离相机最近、最可能是真的）。

---

## 2.8 第七步：数字识别

装甲板框出来后，数字在哪里？在两个灯条的**正中间**。

**ROI（感兴趣区域）怎么裁的：**
```
装甲板四角 → 画一个外接矩形
  → 水平方向：取中间的 35% 宽度（正方形）或 60% 宽度（长方形）
  → 垂直方向：取中间的 85% 高度
  → 这几本覆盖了数字所在的区域
  → 传灰度 → 缩到 32×32 → 归一化 [0,1]
```

**为什么要区分长方形和正方形？** 正方形装甲板数字外面有一个白色的机器人图标，裁多了图标边缘会干扰 AI。所以裁窄一点（35%），只取核心数字区。长方形装甲板没有图标，数字也大，可以裁宽一点（60%）。

**32×32 是怎么来的？** AI 模型（tiny_resnet.onnx）训练时就用的 32×32 的图。我们必须把裁出来的区域缩放到同样大小，模型才能认。

**缩放逻辑：** 保持原来的长宽比例不变，缩到刚好能放进 32×32 的框里。多余的地方填黑色。这一步保证了数字不会被"压扁"或"拉长"。

---

## 2.9 第八步：文字标签

黄框画好后，在框的左上方标注装甲板颜色和数字，比如 `blue_1`。

**为什么文字要用翻转的方式？** OpenCV 的 `putText`（画文字函数）画出来的字，在你的相机画面里方向不对。所以我们采用了"反着画"的技巧：

```
① 在一个空白小图上正常写字
② 把这小图翻转 180°
③ 贴到装甲板左上角
→ 这样在大图上看起来就是正向的了
```

# 第三部分：常见问题速答

**Q：为什么不用一个固定参数适应所有光照？**
A：暗环境和亮环境下，同一个灯条的 HSV 值完全不同。暗环境灯条 S=80, V=120；亮环境灯条偏白 S=20, V=250。目前参数针对暗环境（你的使用场景）优化。

**Q：BGR 过滤和 HSV 过滤有什么区别？**
A：HSV 告诉电脑"大概在蓝色范围就行"。BGR 告诉电脑"蓝色必须远远碾压红绿"。前者宽松，后者精确。

**Q：为什么数字识别有时候闪？**
A：数字识别依赖装甲板检测。如果灯条漏检了→装甲板丢了→数字跟着丢。我们加了"帧间记忆"：数字出现后保留 30 帧，黄框短暂消失了数字还能撑一会儿。

**Q：为什么暗环境能检测，亮环境不行？**
A：参数是为暗环境调的。亮环境需要完全不同的参数（更高的 S 和 V 门槛）。这个留到以后再完善。
